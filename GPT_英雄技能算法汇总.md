# 百将牌英雄技能算法汇总（供 GPT 网页版处理）

## 给 GPT 的处理要求

你正在处理一个三人斗地主英雄技能 AI 项目。请以本文档中的技能注册表为事实来源，不要自行发明英雄、技能、触发条件或次数限制。

处理规则：

1. 保留英雄名、技能名、技能类别、次数上限和效果语义。
2. 区分“规则已登记”“模拟已验证”和“UI 实战已验证”，不要把三者混为一谈。
3. 修改算法时，技能定义放在 `ok_tasks/card_ai/heroes.py`，技能结算放在 `hero_effects.py`，技能选择和牌路评估放在 `hero_policy.py`。
4. 实战出牌入口是 `ok_tasks/RlCardRuleModel.py` 的 `predict()`；屏幕识别和技能按钮交互位于 `ok_tasks/AiCardPlayingTask.py`。
5. 新增或修改技能后，应检查英雄总数、技能总数、唯一 `rule_id`、触发处理器、效果处理器和合法动作守恒。

## 项目算法结构

| 文件 | 职责 |
|---|---|
| `ok_tasks/card_ai/heroes.py` | 全部英雄技能注册表 `HERO_REGISTRY`，以及技能分类、账号英雄名单和验证状态 |
| `ok_tasks/card_ai/hero_policy.py` | 技能选择、技能结算投影、剩余牌路估算、动作评分与最佳动作选择 |
| `ok_tasks/card_ai/hero_effects.py` | 各种技能效果的具体结算处理器 `EFFECT_HANDLERS` |
| `ok_tasks/RlCardRuleModel.py` | 实战规则模型及出牌入口 `predict()` |
| `ok_tasks/AiCardPlayingTask.py` | 屏幕识别、英雄识别、技能弹窗交互和模型调用 |
| `ok_tasks/card_ai/policies.py` | 离线模拟和稳定规则策略 |
| `ok_tasks/card_ai/engine.py` | 完整牌局状态及规则引擎 |

主要调用链：

```text
AiCardPlayingTask
  -> RlCardRuleModel.predict
  -> hero_policy.evaluate_play / select_best_action
  -> heroes.HERO_REGISTRY
  -> hero_effects.EFFECT_HANDLERS
```

## 数据规模和分类

- 英雄总数：65
- 技能总数：102
- 被动技能：49
- 交互技能：36
- 主动技能：17
- 当前账号拥有英雄：24
- 当前账号纯被动英雄：12

类别定义：

- `被动`：满足触发条件后自动结算。
- `交互`：满足条件后还需要选择牌、目标、确认或取消。
- `主动`：由策略主动决定是否发动。
- “不限”表示注册表没有固定次数上限，不表示游戏规则一定没有其他状态条件。

## 技能数据结构

每个技能在代码中使用 `HeroSkillSpec` 表示，主要字段如下：

```text
hero                 英雄名
name                 技能名
trigger              触发处理器名称
limit                次数上限，None 表示无固定上限
effect               效果处理器名称
category             passive / interactive / active
interactive          是否需要二级交互
rule_id              唯一规则编号
rules_version         规则版本
choice_kind           automatic / interaction
optional              是否允许跳过
documented            是否已有规则文档
projection_verified   是否通过技能投影验证
sim_verified          是否通过完整模拟验证
ui_verified           是否通过真实 UI 验证
```

## 验证注意事项

1. 本表中的 65 名英雄都已进入 `HERO_REGISTRY`，但“已登记”不代表全部经过真实牌局 UI 验证。
2. GPT 如果生成代码，必须继续保留 `sim_verified` 和 `ui_verified` 的真实性，不能为了通过测试直接全部设为 `True`。
3. 技能造成的新增牌、复制牌、万能牌必须带来源标记，不能破坏牌实例守恒和历史记录。
4. 所有交互技能都必须提供安全取消或暂停路径；强制交互不能在无法确认界面状态时盲点按钮。
5. 技能评价应比较完整结算后的剩余牌路和最坏风险，不能只按当前牌面点数贪心选择。
6. 修改后至少验证 `tests/TestHeroPolicy.py`、`tests/TestCardAiV3.py` 和 `tests/TestAiCardPlaying.py`。
